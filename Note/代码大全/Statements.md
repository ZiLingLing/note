# Statements 语句
## 第十四章 组织直线型语句
本章介绍最简单的控制流：即按先后顺序放置语句和语句块。
### 14.1 必须有明确顺序的语句
如果语句之间存在依赖关系，并且这些关系要求把语句按照一定的顺序加以排列，那么请设法使得这些依赖关系变得明显。下面是一些用于组织语句的简单原则：
- 设法组织代码，使依赖关系变得明显  
  例如把其他子程序依赖的数据的初始化流程放入一个InitData方法，这样明确了其他子程序对InitData方法的依赖，这个子程序的名字清楚地表明了应该在运行其他计算用途的子程序之前调用它。
- 使子程序名能凸显依赖关系
- 利用子程序参数明确显示依赖关系  
  当所有子程序都用了同一个数据的时候，从中得到的暗示是“它们可能操作了相同的数据”，因此这些语句的顺序可能重要。由此可知，如果子程序所用数据不同（且不是前面子程序的结果作为了后面子程序的输入），那么可能暗示这些语句的顺序可能不重要。
- 用注释对不清晰的依赖关系进行说明  
  在编码时应当依赖于前面提到的那些技术进行自解释和获得好的抽象，但由于一些原因无法改进代码本身时就可以用文档弥补代码的不足。
- 用断言或者错误处理代码来检查关系  
  如果代码非常重要，可以用断言和错误处理在关键处对顺序依赖关系进行说明。
ps：具体例子在书P348-P350

### 14.2 顺序无关的语句
当一条语句并把依赖或者在逻辑上承接另一条语句，但顺序的确对可读性、性能和可扩展维护性有影响，而且缺少执行顺序依赖关系时，可以用第二标准——“就近原则”把相关操作放在一起。
- 使代码易于自上而下地阅读  
  要使得读者易于自上而下阅读，而不是让读者的目光跳来跳去。并且不仅仅满足于控制流运行时自上而下，还要满足阅读时自上而下不用搜索整个子程序来寻找信息。
  ```个人理解
  这可能也就是goto语句不被喜欢的原因吧，它降低了可读性，让读者的视线“跳来跳去”。(说goto语句“危险”、提高了复杂性使得程序更容易出错应当算作别的问题即使很可能是因为可读性差引起的)。
  ```
- 把相关语句组织在一起  
  把相关语句放在一起。一些语句之所以相关，是因为它们都处理了相同的数据、执行了相似的任务，或者有某种执行顺序上的依赖关系。  
  检查相关语句是否组织得好的方法：把相关的子程序用框框上，如果框之间没有重叠那么就说明相关语句组织得很好。  
  ps：这么做之后可能发现每个语句块与前后被框起来的语句块没有多少有意义的联系，此时可以考虑把每个内部关联性很强语句块提取重构成独立的子程序。

  
## 第十五章 使用条件语句
条件语句用来控制其他语句是否执行（即使for、while等循环控制符看作条件语句也是合理的，但传统上还是把它们分别对待）
### 15.1 if语句
- 简单的if-then语句
  写if语句的时候遵循下列原则：
  - 首先写正常的代码路径，再处理不常见情况  
    这么做能够保证不常见的情况不会遮掩正常的执行路径，这对可读性和代码性来说都很重要。
  - 确保对于等量的分支是正确的  
    这个建议的含义，也就是不要出现使用">"替代">="这样的情况，这会在边界条件处出问题，从而犯下off-by-one错误。
  - 把正常情况的处理放在if后面而不要放在else前面（也就是放在if块而不是else块）  
    把你认为会正常的情况放在前面来处理，这符合把决策的结果代码放在尽可能靠近决策位置的一般原则。
  - 让if子句后面跟随一个有意义的语句  
    也就是不要让if语句块为空。
  - 测试else子句的正确性
  - 检查if和else子句是不是弄反了
- if-then-else语句串
  写if-then-else语句串的时候，注意下述指导原则：
  - 利用布尔函数调用简化复杂的检测  
    也就是前文提过的，对于复杂的布尔判断语句，用返回布尔值的函数、起个好的方法名能提高可读性。
  - 把最常见的情况放在最前面  
    这样可以让阅读代码的人为找出正常情况的处理代码而必须要读的处理非常见情况的代码变少，也就是“让阅读者不必为了找到正常流程，先啃完一大堆非正常情况的处理代码”。
  - 确保所有的情况都考虑到了  
    写一个放在最后的else语句，用出错消息或者断言来捕捉那些不考虑的情况。这种情况不是给用户而是给开发者看的。
  - 如果你的语言支持，把if-then-else语句串替换成其他结构

### 15.2 case语句
- 为case选择最有效的排列顺序
  下面是使用case语句的几点提示：
  - 简化每种情况对应的操作  
    简短的情况处理代码会使case语句的结构更加清晰。
  - 不要为了使用case语句而刻意制造一个变量  
    case语句应该用于处理简单的、容易分类的数据。如果数据并不简单，那么就使用if-then-else语句串。
  - 把default子句只用于检查真正的默认情况  
    即使只剩下一种情况需要处理时，也不要将这种情况写入default，因为这会失去case语句的标号所提供的自动说明功能（也就是case后:前的部分）。
  - 利用default子句来检测错误
    如果default子句既没有用作其他处理，按照正常执行顺序也不太可能调用，那么就向里面写入一条诊断消息，这种情况意味着每一个case语句的值都是合法的，所以default语句只需要进行错误检测。
  - 在C++和Java（这类case语句不会自动跳出的语言）里避免代码执行越过一条case子句的末尾  
    因为不会自动跳出，避免“聪明的”不适用break语句增加阅读困难，因为它把不同的控制结构搅在一起了。
  - C++里，case末尾明确无误地标明需要穿越执行的程序流程  
    如果故意要越过某一case语句的末尾，起码要在相应的位置给出注释解释为什么要这样编写代码。

## 第十六章 控制循环
“循环”是一个非正式的术语，用来指代任意一种迭代控制结构——任一能够导致程序反复执行一段代码的结构。
### 16.1 选择循环的种类
循环的种类：  
1.计数循环。执行次数一定。  
2.连续求值循环。会在每次迭代时检查是否该结束。  
3.无限循环。例如游戏的主循环。  
4.迭代器循环。对容器类里面的每个元素执行一次操作。  

灵活度和检查位置决定了如何对用作控制结构的循环种类进行选择：  
灵活度：
- 严格——循环执行的次数是一定的,
- 灵活——在每次迭代的时候检查循环有没有完成  

检查循环是否执行完毕的位置：  
1.循环开始处，循环中的代码不一定被执行。
2.中间，检查之前的代码一定被执行。
3.结尾处，代码至少执行一次。

- 什么时候使用While循环  
While循环是一种灵活的循环选择。如果预先不知道循环迭代的次数，那么就用while循环。  
  有关while循环最主要的事项就是决定在循环开始做检测还是在循环结尾做检测。
  - 检测位于循环的开始  
    循环可以执行任意次数时使用。绝大多数语言都包含了while循环。
  - 检测位于循环的结尾  
    需要一个灵活的循环，但该循环至少需要执行一次。这时候用do-while循环，或者在不支持该语句的语言里模拟在末尾进行条件检测的循环。
- 什么时候用带退出的循环  
  带退出的循环就是终止条件出现在循环中间而不是开始或者末尾的循环。也就是用while和break模拟它，或者用goto。
  - 正常的带退出的循环  
    使用场景：当判断条件必须在循环途中拿到计算结果（产物）才能做出时。（从循环开始处进行第一次迭代）  
    使用带退出的循环需要注意的细节：
    1.把所有的退出条件放在一处。这样有助于提高可读性，并且在修改时可以防止遗漏推出条件从而减少犯错。  
    2.用注释来阐明操作意图。
  - 非正常的带退出的循环  
    使用goto之间进入到循环的中间，从循环中间开始（第一次）迭代，然后迭代到不满足循环条件为止退出。

- 何时使用for循环
  需要一个执行固定次数的循环时使用for循环。  
  使用for循环注意事项：  
  - 不要在内部修改下标影响循环。（如果需要修改下标值迫使循环终止，那么证明这个场景更适合while循环）
  - 在头部写好循环条件后，内部无需关注循环控制问题，无须在循环内部做任何事情终止它。

- 何时使用foreach循环
  foreach循环或其等价物很适用于对数组或者其他容器的各项元素执行操作。优势在于消除了循环内务处理算术，从而也就消除了任何由循环控制算术导致出错的可能性。

### 16.2 循环控制
循环语句会出现的错误的类型：
- 忽略或错误地对循环执行初始化
- 忽略了对累加变量或其他与循环有关的变量执行初始化
- 不正确的嵌套
- 不正确的循环终止
- 忽略或者错误地增加了循环变量的值
- 用不正确的循环下标访问数组元素

阻止上述错误发生的方法：
- 减少能影响该循环各种因素的数量，不断地简化循环。
- 把循环内部当做一个子程序看待——把控制尽可能地放到循环体外。
  把循环条件表达清楚。把循环当作一个黑盒，外围程序只知道它的控制条件，而不应该知道它的内容。
  ```个人理解
  循环内部尽量用自己的局部变量，少用作用域比循环语句块大的变量。
  ```

- 进入循环
  在代码进入循环时使用下述原则：
  - 只从一个位置进入循环  
    每次都要从循环头处进入，不要像“非正常的带退出的循环”一样通过goto进入循环（真的很危险，而且代码读起来令人费解）。
  - 把初始化代码紧放在循环前面  
    就近原则主张把相关的语句放在一起，这样提高可读性减少因修改时遗漏某处产生的错误。因此把循环的初始化语句和与之相关的循环放在一起是合理的。（如果放到一个更大的中作为嵌套的循环，里面语句很多很长，初始化语句和内层循环距离远就可能导致修改、复制、移动时忘记初始化条件从而导致错误）。
  - 用while(true)代表表示无限循环  
    主要是不要尝试用一个很大的值作为for循环的边界假装它是个无限循环，因为有终止条件就模糊了循环的用意，降低了可维护性。
  - 在合适的情况下多用for循环  
    for循环把循环的控制代码集中在一处，有助于写成可读性强的循环。
  - 在while循环更适用的时候，不要使用for循环  
    因为for循环的循环头里可以写与控制完全无关的代码。应该只把for循环的循环头的位置保留给循环控制语句。

- 处理好循环体
  - 用“{”和“}”把循环中的语句括起来  
    明确作用域有助于提高可读性。最好任何时候（即使是python这样不要求一定使用正反花括号的语言）都用正反花括号明确作用域。
  - 避免空循环  
    避免循环内部没有语句，功能在控制时执行。（也就是前面说的不要写控制无关的代码）例子在P375下面部分。
  - 一个循环只做一件事  
    除非区分很多循环会使得运行效率低下。

- 退出循环  
  如何处理循环尾：
  - 设法确认循环能够终止
  - 使循环终止条件看起来明显
  - 不要为了终止循环而胡乱改动for的下标。
  - 避免出现依赖于循环的下标最终取值的代码  
    不要在循环结束后继续使用循环的下标。
  - 考虑使用安全计数器  
    为了避免死循环。每次运行后加一递增安全计数器，如果执行次数过多就结束循环。但安全计数器不应该被滥用，它的存在本身就增加了复杂度导致更容易出错，例如忘记维护引发的错误。仅在关键的循环中使用即可。

- 提前退出循环  
  continue不会让程序从循环退出，而是让程序跳过循环体的余下部分，从该循环的下一次迭代的开始位置继续执行。break会直接退出循环。
  - 考虑在while循环中使用break语句而不用布尔标记  
    这能增加可读性。
  - 小心那些break散布其中的循环  
    一个含有大量break的循环可能意味着编写它的程序员对该循环的结构或对循环在围绕它的代码中的角色缺乏清晰的认识。在大量使用break的场合，用一系列循环而非一个含有多个出口的循环可能会使得表达更清晰。
  - 在循环开始处用continue进行判断  
    可以避免整个循环内部放在if语句块里面。
    ```个人理解
    这种情况使得嵌套层次变深，增加了理解难度，不如把这个if的判断条件取非，然后用continue提前跳出，就减少了一层嵌套。
    ```
  - 使用break和continue时要小心谨慎  
    break（continue）消除了把循环看作黑盒的可能性（因为要理解跳出/跳过条件，就必须了解这部分的实现）。很难说使用break和continue是好还是坏，众说纷纭。建议只有在证明使用break或者continue具有正当性时才使用。

- 检查端点  
  一个简单的循环通常要注意三种情况：开始的情况，任意选择的中间的情况，最终的情况。在创建循环时要确认不会出现任何off-by-one错误（确保不会出现越界错误），如果有些特殊情况是与第一次和最后一次都不同时，也要检查它。用头脑模拟计算和手工运算验证（循环里的运算）都是很好的锻炼自己能力的方式。

- 使用循环变量  
  下面是一些使用循环变量的指导原则：
  - 用整数或者枚举类型表示数组和循环的边界  
    总之不要用浮点数，前面也说过，浮点数运算并不一定准，如果数量相差过大会出现偏差，这种偏差可能导致死循环。
  - 在嵌套循环中使用有意义的变量名来提高其可读性
  - 把循环下标变量的作用域限制在本循环内  
  - 即使编译器允许内部声明一个和循环下标同名的变量，也不要这么做，不要依赖编译器检查，一旦出错会导致很严重的问题（而且看起来还不太容易查）。不要这么做避免用错下标、避免下标串话。
 
- 循环应该有多长  
  循环的长度可以用代码行数或者嵌套层次来衡量。下面给出一些指导原则：
  - 循环要尽可能地短，以便一目了然
    一个显示器屏幕的能容纳的行数就是循环的最大长度。
  - 把嵌套限制在三层以内
    如果超出了三层，就把某一部分提取为子程序或者简化控制结构来缩短它（在概念上）。
  - 把长循环的内容移到子程序里
  - 要让长循环格外清晰
    对于长循环，如果担心给阅读者带来不便，那么就写一个单一出口的循环，并且保持退出条件清晰无误。

### 16.3 轻松创建循环——由内而外
使用一种简单的技巧使它一开始就是对的。下面是一般处理过程。从一种情况开始，用字面量来编写代码，然后缩进它，在外面加上一个循环，然后用循环下标或者计算表达式替换那些字面量。如果需要就继续在外面套上一层循环，再替换掉一些字面量。根据需要持续这一过程。等到做完后加上所有初始化。这个过程就是从嵌套最深的循环开始依次向外生成代码的。

### 16.4 循环和数组的关系
循环的下标和数组的下标一一对应，循环就是用来操作数组的。但循环和数组并非天生就互相关联，有些语言提供强大的数组操纵功能从而无需依赖循环且更不容易出错。所用语言在相当程度上影响到解决方案。

## 第十七章 不常见的控制结构
### 17.1 子程序中的多处返回
下面给出一些使用return语句的指导原则：
- 如果能增加可读性，那么就使用return  
  某些子程序里一旦知道了想要的答案，会希望马上返回到调用方子程序。比如针对于计算时的特殊情况直接知道了结果从而直接返回，这样就不必继续阅读无用代码。
- 用防卫子句（早返回或早退出）来简化复杂的错误处理  
  避免过深的嵌套（大于等于4层），把深层次的if嵌套拆分成并列的浅层次的并列的判断异常的条件分支语句，针对每个异常都提前return。
- 减少每个子程序中return的数量  
  只在能增加可读性的时候使用多处返回。判断的诀窍是“如果你读到子程序的后部时，没有意识到它从前面某处返回的可能性”时就不宜用，因为不仅它没有帮助你理解代码，反而引入了更多的复杂度。

### 17.2 递归
在递归里面，一个子程序自己负责解决某个问题的一小部分，它还把问题分解成很多的小块，然后调用自己来分别解决每一小块。当问题的小部分很容易解决，而问题的大部分也很容易分解成众多的小部分时，常常会用到递归。
```个人理解
递归就是“拿一把没完成的锤子做那个待完成的锤子”。
```
对于某一小范围内的问题,使用递归会带来简单、优雅的解。在稍大一些范围里，使用递归会带来简单、优雅但是难懂的解。对于大多数问题，它所带来的解将会是极其复杂的——在那些情况下，使用简单的迭代通常会比较容易理解。因此要有选择地使用递归。

- 使用递归的技巧  
  使用递归时需要记住以下技巧：
  - 确认该递归能停止  
    检查子程序以确认其中含有一条非递归路径。编写递归子程序的关键目标之一就是需要避免无穷递归。
  - 使用安全计数器防止出现无穷递归  
    在不允许使用上述简单测试的环境中使用递归时，需要用安全计数器防止产生无限递归。
  - 把递归限制在一个子程序内  
    循环递归（A调用B，B调用C，C调用A）非常危险，因为很难检查。
  - 留心栈空间  
    用了递归之后将无法保证程序会使用多少栈空间，也就很难预测程序在运行期间会表现得怎样。不过可以按照下述步骤来控制程序在运行期间的表现：  
    1.如果使用了安全计数器，那么在给它设置上限时需要考虑的事项之一就是，愿意给递归子程序分配多少栈空间，要把它设置得足够低，以防止栈溢出。  
    2.应注意观察递归函数中局部变量的分配情况，特别要注意内存消耗大的对象。要用new在堆上创建对象，而不要让编译器在栈上面自动创建对象。
  - 不要用递归去计算阶乘或者斐波那契数列  
    作者认为计算机教科书上用阶乘或者斐波那契数列讲解递归这件事很蠢。除了速度缓慢，并且无法预测内存使用状况以外，用递归写出的子程序比用循环写出这两个例子的子程序更难理解。  
    在递归之前应该考虑替换它的方案，递归都可以用循环或者栈来替换。

    ```个人理解
    使用递归是否是个好方案，取决于问题本身有没有自相似结构。阶乘和斐波那契没有——它们是线性递推，循环最直观。树、图、分治、回溯、嵌套解析有，递归比循环更接近问题本身的样子。
    此处的“自相似”我认为是每一个计算单元会分叉出多个子调用，结构像一棵倒挂的树，就像分形一样，其侧重于“多个”；而线性递推的每一步只依赖前一步，调用链是一条直线往下走，没有分叉。
    ```
