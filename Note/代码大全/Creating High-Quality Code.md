# Creating High-Quality Code
## 第五章 软件构建中的设计
### 5.1 设计中的挑战
- 设计是一个险恶的问题
  - “险恶的问题”指只有通过解决或部分解决才能被明确的问题，即把这个问题“解决”一遍才能够明确定义，才能再次解决从而形成一个可行的方案。
- 设计是个了无章法的过程（即使得到的成果是清爽的）
  - 在设计的过程中会犯很多错，在设计阶段修改错误是设计的关键所在。
  - 优、劣设计之间的差异往往非常微妙。
  - 无法判断设计何时算“足够好”。设计永无止境。
- 设计就是确定取舍和调整顺序的过程
  - 设计者工作的一个关键内容便是去衡量彼此冲突的各项设计特性，并尽力在其中寻求平衡。比如程快速的反应速度和缩减开发时间之间的冲突，或者内存和运行时间的冲突等。
- 设计收到诸多限制
  - 设计的要点，一部分是在创造可能发生的事情，另一部分又是在限制可能发生的事情。设计是在有限的资源中寻找最好的方案。
- 设计是不确定的
  - 设计同样一套程序，可能会出现很多种截然不同的设计方法。
- 设计是一个启发式的过程
  - 设计的过程充满不确定性，设计技术也就趋于探索性而不是保证能产生预期结果的可重复过程。因为没有任何工具是完全在各个方面通用的。
- 设计是自然而然形成的
  - 把上面的特性综合归纳起来看，设计是在不断地设计评估、非正式讨论、些试验代码并修改试验代码中演化和完善的。

### 5.2 关键的设计理念
- 软件的首要技术使命：管理复杂度
  - 偶然的难题和本质的难题
    这两类不同的问题导致软件开发变得困难。
    
    偶然的属性指的是一件事物碰巧具有的属性，有没有这些属性都不影响这件事物本身，是一些附属的、随机的、非必要的或者偶然出现的性质。且这些问题大多已经被解决。
    
    本质上的困难来源于复杂性，究其原因是软件开发就是不断地去发掘错综复杂、相互连接的整套概念的所有细节的过程。必须去面对复杂无序的现实世界、精确完整地识别各种依赖关系和例外情况、设计出完全正确而非大致正确的解决方案等。
  - 管理复杂度的重要性
    软件的首要技术使命便是管理复杂度。当项目由技术因素导致失败时，其原因通常是失控的复杂度，有关的软件变得极端复杂，当没人能知道对一处代码的改动会对其他代码带来什么影响时，项目也就快停止进展了。
    
    我们不应该在同一时间把整个程序都塞进自己大脑，应该以某种方式去组织程序，以便能够在同一时刻只专注于一个特定的部分（读这段话时的想法：从结构化编程到面向对象编程也许是遵从了这个思想？）。
此时出现bug的原因则是漏掉了整个程序的某一个部分，

    在软件架构的层次上通过把整个系统分解为多个子系统降低问题的复杂度，保持每个子系统功能的简单，且减少子系统间的相互依赖，子系统间相互依赖越少越容易在同一时间专注问题的一小部分。
保持子程序的短小精悍也能帮助减少思考负担。从问题的领域着手而不是从底层实现细节入手编写程序，在抽象层次上工作也能减少负担。
  - 如何应对复杂度
    高代价、低效率的设计的根源：
      1.用复杂的方法解决简单的问题。
      2.用简单但错误的方法解决复杂的问题。
      3.用不恰当的复杂方法解决复杂问题。
    用下面的方法管理复杂度：
      1.把任何人在同一时间需要处理的本质复杂度的量减到最少。
      2.不要让偶然性的复杂度无所谓地快速增长。
- 理想设计的特征
    - 最小的复杂度。避免在不必要时用“聪明的”设计（例如在没有相关需求时用上精妙却一时难以理解的算法，或者滥用设计模式），因为这些设计通常难以理解。
    - 易于维护。例如自解释的变量名、方法名等。
    - 松散耦合。让程序的各个组成部分间的关联最小。通过应用类接口中的合理抽象、封装性及信息隐藏等原则设计出相互关联尽可能少的类。会减轻测试维护时的工作量。
    - 可扩展性。能增强系统功能而无须破坏其底层结构，能改动某一部分而不影响其他部分。
    - 可重用性。
    - 高扇入。让大量的类使用某个给定的类，如工具类。
    - 低扇出。让一个类里少量或适中地引用其他类。高扇出（大于7个）说明一个类大量使用了其他类，可能使得其变得过于复杂。对于子程序和类而言，低扇出原则都是有利的。
    - 精简性。设计出的系统没有多余的部分，因为任何多余的代码都需要开发、复审和测试，并且在修改其他代码后还需要重新考虑它们的兼容。
    - 层次性。尽量保持系统各个分解层的层次性，使得开发人员能够在任意的单一的层面上观察系统。这样一是可以把低劣的代码屎山禁闭起来不扩散，二是在抛弃或者重构代码时不必修改除了交互层外的任何新代码。
    - 标准技术。开发时的统一标准。
- 设计的层次
  区分层次的原因：某种技术并非适用于所有的层次。
  - Software System 系统层
    - 整个子系统
  - Division into Subsystem or Packages 分解为子系统或者包
    - 把整个程序分为主要的子系统（需要对每个子系统内部选用恰当的方法），并定义清楚允许各子系统如何使用其他子系统（通过限制子系统之间的通信能够让每个子系统更有存在意义）。本书认为对于十几天以上的项目很适合。
      如果所有子系统能同其他子系统通信，那么就完全失去了把它们分开带来的好处。个人的理解是这样会使得依赖关系变得复杂难以理解、必须同时部署多个模块才能保证功能正常、逻辑上划分的模块边界在运行时被频繁穿透使得模块的独立性丧失、使得系统脆弱导致一个子系统的故障或变更可能迅速传播到其他所有子系统。
      系统层的设计图应当是无环图。
  - Division into Class 分解为类
    - 在这一层次上的设计包括识别出系统中所有类。本书认为对于超过几天的项目就需要把子系统分解为类。如果项目很小，可以直接从第1层直接进入到第3层的类视图。
      定义子系统中的类时，也就同时定义类这些类于系统的其余部分打交道的细节，尤其要确定好类的接口。
      这一层的主要设计任务把所有子系统进行适当的分解，并确保分解出的细节都恰到好处，能够用单个类实现。例如对于数据库接口子系统可能被进一步划分为数据访问类、持久化框架类以及数据库元数据。
  - Division into Routines 分解为子程序
    - 这一层的设计细化出类的private子程序，通常是子程序都很简单，但也有很多子程序由更多层次化阻止的子程序所组成。
  - Internal Routine Design 子程序内部的设计
    - 为每个子程序布置详细的功能，即包括编写伪代码、选择算法、组织子程序内部的代码块，以及用编程语言编写代码。
   
  ### 5.3 设计构造块：启发式方法
  对于具体的编程工作来说，期待确定性的行为是正常的；但对于软件设计来说，因为软件设计是非确定性的，因此灵活熟练运用一组有效的启发式方法“指引试错”有时能产生优秀的结果。
  - Find Real-World Objects 找出现实世界中的对象  
    首选且最流行的方法，面向对象设计。设计步骤（无须以特定顺序完成）：
    - 辨识对象及其属性  
      计算机程序通常基于现实世界实体。基于现实世界中对应的实体以及之间的关系画出面向对象视图，辨认对象的特征并将其作为属性。即定义类和类的属性。
    - 定义可对对象执行的操作  
      根据现实世界实体可执行的操作，定义该对象的操作。即定义方法。
    - 确定每个对象可以对其他对象进行的操作  
      由对象间的交互关系确定。例如两种最常见的关系——包含和继承，哪些对象可以包含其他对象，哪些对象可以从其他对象继承。更复杂的系统也会包含更多其他的交互关系，例如关联、聚合、组合等。
    - 确定对象的哪些部分对其他对象可见  
      对数据和方法进行区分，找到哪些是public，哪些是非公有的。
    - 定义每个对象的接口  
      使用编程语言编写完成接口。对象对其他对象暴露的数据及方法都被称为该对象的“公开接口”，通过继承关系向派生对象暴露的部分被称为“受保护的接口”。  
  - From Consistent Abstractions 形成一致的抽象  
    抽象的定义：一种能让你在关注某一概念的同时可以放心地忽略其中一些细节的能力——在不同的层次处理不同的细节。  
    抽象的主要好处在于使得开发人员能忽略无关的细节，即“从高层的细节来看待一个对象”。优秀的程序员会在子程序接口的层次上、类接口的层次上以及包接口的层次上进行抽象。  
  - Encapsulate Implementation Details 封装实现细节  
    封装帮助管理复杂度的方法是“从对象之外的其他地方看不到这些复杂度”。  
    封装填补了抽象留下的空白，是对抽象的补充。抽象只是忽略低层次的实现细节，而封装意味着看不到对象任何其他的细节。  
  - Inherit-When Inheritance Simplifies the Design 当继承能简化设计时就继承   
    继承的好处在于能够很好的辅佐抽象的概念，继承本身就进行了抽象和分层，把对象的共同点放在基类中，而不同点则放在继承基类的派生类中。
    继承能够简化编程的工作，原因就是能够写一个基类来定义共同属性并处理相同的基本操作，另写一些特定的子类来处理特定类的特定操作。  
  - Hide Secrets(Information Hiding) 隐藏秘密(信息隐藏)
    - Secrets and the Right to Privacy 秘密和隐私权  
        当信息被隐藏后，每个类（或者包和子程序）都代表某种对其他类保密的设计或构建决策。隐藏起来的可能是某个易变区域，或者某种文件格式，或某种数据类型的实现方式，或某个需要隔离的区域，这个区域中的发生的错误不会给程序的其余部分带来太大的损失。（此处想到一个简单的例子，例如“++count”这个逻辑在某个类里被广泛使用，此时将它封装为一个方法会更好，倘若需求发生变更，只需要改这一处，减小改错成本）。  
    - An Example of Information Hiding 信息隐藏的一个例子  
        隐藏设计决策对于减少“改动所影响的代码量”而言是至关重要的。  
        此处书上举的第一个例子同上一部分中“封装++count逻辑”想表达的意思相同，此处建议将id = ++g_maxId封装为NewId()方法。
        第二个例子从第一个例子延伸，当需要改变ID类型时，比如将id从int类型改为string类型，而此时已经大量使用了int id = NewId()，如果要进行修改仍然会很繁琐。避免的方法是隐藏ID的类型，自定义一个IdType类型，将这个类型实现和int相同的操作，以此解释int类型，从而避免直接使用，此时如果要进行修改，修改范围只需要限定在IdType类内部即可。  
    - Two Categories of Secrets (隐藏的)两种秘密  
        信息隐藏所说的秘密主要分为两大类：  
          1.隐藏复杂度。不需要应付它，除非在需要关注这部分的时候。  
          2.隐藏变化源。当变化发生时，影响能被限制在局部范围内。复杂度的根源包含复杂的数据类型、文件结构、布尔判断以及晦涩的算法等等。  
    - Barriers to Information Hiding 信息隐藏的障碍  
          1.信息过于分散。例如有一个到处被调用用于存储信息的数组，解决办法也很简单，用子程序隐藏，外部调用子程序，此时只有子程序清楚实现细节。  
          2.循环依赖。循环依赖的问题就是它会使得系统难于测试，互相依赖的两部分无法被单独测试。  
          3.把类内数据误认为全局数据。全局数据通常面对两类问题：一种是子程序在全局数据上进行操作，但是数据可能同时被其他子程序操作，这会导致数据被污染；另一种是子程序知道其他子程序同一时间也在对全局数据进行操作，但是子程序不知道进行了哪些操作。类内数据与全局数据不同，对类内数据，子程序不仅能够知道数据是否被其他的子程序的操作，还能知道其他子程序进行的具体操作，这些东西在类内是透明的（前提是类的设计足够精巧，如果类过于臃肿也会面临全局数据面临的问题）。  
          4.可以察觉的性能消耗。在架构层上按照信息隐藏的目标设计系统并不会与按照性能目标设计冲突。即使在编码层上也不用担心因为子程序调用和间接访问会带来性能上的损耗，因为在编码层上能够衡量系统性能，并找出妨碍性能的瓶颈，为了找出妨碍性能的瓶颈，应当做出高度模块化的设计，这样在之后尝试突破瓶颈时不会对系统的其他部分产生影响。
        
    - Value of Information Hiding 信息隐藏的价值    
          从信息隐藏原则思考设计，能做出更好的设计决策。例如隐藏变量类型，例如隐藏类内部的数据通过接口给出数据，例如使得系统层上做出更好的类和子系统分解和交互的决策。  
  - Identify Areas Likely to Change 找出容易改变的区域  
      这里是将上文反复提及的代码分离相关事情专门拿出来讲。
      
      应对需求变化的能力对代码编写的重要性再次不再重复。应采取的应对变动的措施：  
        1.找出看起来容易变化的项目。如果需求包含了潜在变化的清单，以及其中每一项变化发生的可能性，这种情况下就很容易找出潜在的变化。  
        2.把容易变化的项目隔离开。把容易变化的部分单独或者和其他易变化组件划分为一个类。  
        3.把看起来容易变化的项目隔离开来。设法设计好类之间的接口，使其对潜在变化不敏感。即前文反复提及的“将易变部分隐藏”。
      
      下面是一些容易发生变化的区域：  
        1.业务规则。最容易变动的地方。  
        2.对硬件的依赖性。例如屏幕、打印机、键盘、鼠标、硬盘、声音设施以及通信设计等之间的接口。把它们隔离在自己的子系统和类中利于把程序移植到新的硬件环境下。  
        3.输入和输出。  
        4.非标准的语言特性。没有被官方标准化组织（如 ECMA、ISO、W3C 或 WHATWG）正式写入规范文档，但被某个特定供应商（如浏览器厂商、编译器作者）实现并提供的功能。  
        5.困难的设计区域和构建区域。如果做得差会修改。  
        6.状态变量。一开始可能是bool，随着状态变多则需要改变，因此最好一开始就用枚举，然后通过访问子程序即方法获取状态。  
        7.数据量的限制。例如分配一个数组，数组的大小多数情况下对于其他地方是无效信息，可以用具名常量隐藏。

      如何寻找容易发生变化的区域：  
        首先找出程序中可能对用户有用的最小子集，这是系统的核心，是不容易发生改变的区域。自这个最小子集开始扩展系统，在扩展的过程中潜在的改进区域就变成了易变化的区域，例如把程序变成线程安全的，加入系统的本地化，在增量部分的设计中提取易变化的部分做好信息隐藏。

  - Keep Coupling Loose 保持松散耦合
    耦合度表示类与类或者子程序与子程序之间关系的紧密程度，耦合度设计的目标是创建出小的、直接的、清晰的类或者子程序。保持模块之间的关系尽量简单，尽量少或者不依赖其他模块。
    - Coupling Criteria 耦合标准  
      - 规模。此处规模指模块之间的连接数。被调用的类中public方法越少，调用的方法需要传入的参数越少，则耦合越松散。  
      - 可见性。指两个模块间连接的显著程度。相比于用修改全局变量再调用其他模块，传入参数的做法会更好。  
      - 灵活性。指模块之间的连接是否容易改动。灵活性是其他的耦合特性综合作用的结果。总之一个模块更容易被另一个模块调用，使用时的限制条件越少（例如使用场景的限制、传入参数的类型和数量以及含义），灵活性越高，也越容易维护。  
    - Kinds of Coupling 耦合的种类  
      - 简单数据的耦合。两个模块间参数传递数据，且数据类型是简单数据类型（例如C#中除开结构体的值类型，字符串之类的）。简单数据的耦合是可以接受的。
      - 简单对象的耦合。如果一个模块实例化一个对象，那么它们之间的耦合关系就是简单对象耦合的，这种耦合关系也很不错。
      - 对象参数耦合。如果传参是对象，相比传简单数据类型耦合会更紧密一些，因为它要求被调用的方法了解对象含有的各种信息（如字段、方法）。
      - 语义上的耦合。最难缠的耦合关系。一个模块不仅调用了另一个模块的语法元素，而且还使用了有关那个模块内部工作细节的语义知识。语义耦合发生时，被调用的模块中的代码如果发生了改变会使得调用模块的功能变得不符合预期，并且这个错误编译器无法检查。  
    松散耦合的关键之处在于一个有效的模块提供一层附加的抽象，使用这个附加的抽象能使得开发人员在同一时间只关注一件事。如果对一个模块的使用要求同时关注好几件事如内部工作细节、对全局数据修改、不确定的功能点，那么模块丧失；了抽象能力，模块所具有的管理复杂度的能力也被削弱甚至完全消失。

## 第六章 可以工作的类
### 6.1 类的基础：抽象数据类型（ADT）
抽象数据类型的定义：一些数据以及对这些数据进行操作的集合。这些操作既向程序的其余部分描述了这些数据是怎么样的，也允许程序的其余部分改变这些数据。
“抽象数据类型可以让你在现实世界中一样操作实体，而不必在底层上摆弄实体。”（这是本书对抽象数据类型的特点的解释）  

抽象数据类型的特点：
- 抽象数据类型是一种数学模型，用于描述数据类型的抽象特征、定义和操作集合，而不关心数据的存储方式或具体实现。
- 抽象数据类型将数据类型的逻辑结构和操作封装在一起，提供了一个抽象层次，使得用户可以独立于具体实现来使用数据。
- 具体的数据结构可以实现一个或多个抽象数据类型，提供了对应的操作和语义。
- 抽象数据类型通常都有对应的数学概念。

- Example of the Need for an ADT 需要用到ADT的例子
  假如一个程序的一部分功能是控制文本的字体，使用不同的字体、字号和文字属性（如粗体、斜体）来控制屏幕上的文本。
  如果使用一个ADT，就能有捆绑在相关数据上的一组操作字体的子程序，包括字体名称、字号和文字属性等，这些子程序和数据集合为一体，就是一个ADT。
  如果不使用ADT，只能使用拼凑的方法操作字体了，例如currentFont.size = 16或者currentFont.size = PointsToPixels(12)，然后再单独使用currentFont.bold = true等其他代码直接控制数据成员，且程序中的很多地方都会充斥着这种直接操纵数据成员的代码。  

- Benefits of Using ADTs 使用ADT的益处  
  就上面的例子，ADT相比于逐个修改字体数据类型并且拼凑在一起，显然会好很多。下面是具体的好处。
  - 可以隐藏实现细节。保护其余部分程序不受改变的影响。
  - 改动不会影响整个程序。如果想要让字体变得更丰富，或者支持更多操作。只需在一处修改，且不会影响到程序的其余部分。
  - 让接口提供更多信息。例如currentFont.size = 16这条语句不够明确，原因在于“16”的单位不明确，可能是像素也可能是磅，上下文并不会提供这些单位（除非有注释等手段），把相似操作集成到一个ADT里就能够基于同一个单位定义整个接口，避免单位混淆。
  - 更容易提高性能。更容易编写出一些更好的子程序，从而不用来回修改。（此处虽然没有明确解释，但前文反复提及抽象、封装的对降低复杂度重要性）  
  - 让程序的正确性更显而易见。封装起来调用方法比有助于缩小错误的范围，对于直接赋值，错误的可能性包含了写错变量、数据，而使用方法排除了这一点。  
  - 让程序更具有自我说明性。通过方法进行赋值，通过明确的、良好的方法命名可以让代码自解释，而直接使用数据、属性赋值无法很好地做到。  
  - 无须在程序内到处传递数据。ADT中可以用一个内部的私有数据类型来保存这些本来只有ADT关心的数据。  
  - 能够用真实世界在的概念来操作，不必在底层实现上操作。这样能给其他部分程序提供更好的抽象层。

- ADT和类
``` 个人的理解
  ADT=数据+对该数据的操作集合。也就是说没有没有ADT的类就像是一堆Getter和Setter的集合体，只记录数据而非抽象行为。抽象数据类型构成了“类”这一概念的基础，在支持类的变成语言里可以把抽象数据类型用类实现，考虑类时，把类看作抽象数据类型加上继承和多态两个概念。
```

### 6.2 良好的类的接口
- 类的接口应该展现一致的抽象层次。需要准确捕捉类的接口的当前所需抽象，并且暴露同一抽象层次同一抽象数据类型的接口。在后续扩展和添加接口的时候也要考虑是否抽象层次相同，如果不同则尽量换一种方式实现
``` 关于如何判断是否同一抽象层次的个人理解
将类暴露出的接口分类，归类成底层对数据的组织、中层对数据的操作（Manager），上层的具体业务（直观表现），如果出现混淆则接口不属于同一抽象层次
``` 
- 尽可能让接口可编程：一个接口中任何无法通过编译器强制实施的部分，就是一个可能被误用的部分，要尽量把语义接口（通过注释、文档等手段说明用法）元素转换成编程接口（编译器检查并强制实施）的元素
- 尽可能地限制类和成员的可访问性。实际上目的还是保护接口抽象的完整性，即如果修改后需要暴露的新的接口不会破坏“同一抽象层次”，则做法才是可行的。
- 不要公开暴露成员数据
- 警惕从语义上破坏封装性

### 6.3 有关设计和实现的问题
- 包含  
  概念：表示一个类含有一个基本数据元素或对象
  - 实现一个包含关系
    把包含想象成拥有某个数据成员的关系
  ```举个例子
  一个类包含基本数据元素int ID, string name
  或者
  把一个对象当作另一个对象的成员，如让一个类中包含public class Employee{ private int ID; private string name; }的实例
  ```
  - 如果是C++，在无法通过“把一个对象当作另一个对象的成员”时，可以使用私有继承（私有继承是包含即has a关系）  
    但继承这会破坏封装性。
  - 警惕数据成员数量超过7个的类
    对于简单数据类型居多的类，这个标准最多可以放宽到9个，对于复杂对象多的类，应该把标准收紧到5个。  
    对于数据成员超过7个的类，考虑把数据成员分解为几个更小的类。
- 继承  
  概念：一个类是另一个类的一种特化。
  - 决定使用继承时的注意事项
    - 公有继承实现“is a”关系。  
      通过继承创建一个新类时表明新类是某个现有类的一个更特殊的版本，如果派生类不完全遵守由基类定义的同一个接口契约，就不该使用继承应转而使用“包含”。
    - 使用继承要么进行详细说明，要么就不使用它。
      如果某个类未被设计为可继承，那么应当把成员定义为非virtual、final（Java）、非override这样无法被继承的状态。
    - 遵循里氏替换原则。
      定义：对于基类中定义的所有子程序，用在它的任何一个派生类中时的含义都应当是相同的。即派生类必须能通过基类的接口调用，且派生类和基类相同接口调用的语义应当相同。  
      继承会提升程序的复杂度，如果遵循里氏替换原则能够降低复杂度，继承能成为降低复杂度的强大工具，因为程序员不再需要思考不同派生类的实现在语义上的差异。
    - 确保只需要继承需要继承的部分
    - 把共用的接口、数据和操作放到继承树中尽可能高的位置
    - 只有一个实例的类是值得怀疑的
      此时应当考虑派生类的差距是否能用数据而非新的派生类表达。
    - 只有一个派生类的基类也值得怀疑
      没有提前设计的必要，不要创建任何并非绝对必要的继承结构。
    - 避免让继承结构过深
      继承的层次最好不要超过2-3层，派生类的数量最好不要超过7个（上下不波动超过2）。
    - 尽量使用多态，避免大量的类型检查
    - 让所有数据都是private而非protected
      如果派生类需要访问基类的属性，应当提供protected访问器方法。

  - 使用继承的规则
    - 如果多个类共享数据而非行为，应该创建这些类可以包含的共用对象。
    - 如果多个类共享行为而非数据，应该让它们从共同的基类继承而来，并在基类里定义共用的子程序。
    - 如果多个类既共享数据也共享行为，应该让它们从一个共同的基类继承而来，并在基类里定义共用的数据和子程序。
    - 当你想由基类控制接口时，使用继承；当你想自己控制接口时，使用包含。
   
 - 成员函数和数据成员
   下面就有效地实现成员函数和数据成员给出建议：
   - 让类中子程序的数量尽可能少
   - 禁止隐式地产生你不需要的成员函数和运算符
   - 减少类所调用的不同子程序数量
   - 对其他类的子程序的间接调用要尽可能少（迪米特法则）
     迪米特法则举例：A对象可以任意调用它自己的所有子程序，如果A对象创建了一个B对象，它也可以任意调用B的public子程序，但应该避免再调用由B对象所提供对象中的子程序。
- 构造函数
  - 尽可能早构造函数中初始化所有数据成员（容易实现的防御性编程）
  - 单例模式下构造函数应用private修饰符
  - 拷贝构造应使用深拷贝，除非论证可行才使用浅拷贝
    
  - 创建类的理由：
    - 对现实世界中的对象建模
    - 对抽象对象建模
    - 降低复杂度
      通常通过信息隐藏达成这一目的。  
    - 隔离复杂度
      修正代码时不易影响其他部分。  
    - 隐藏实现细节
    - 限制变化所影响的范围
      能把变动带来的影响限制在一个或者少数几个类里。  
    - 隐藏全局数据
    - 让参数传递更顺畅
      如果需要把一个参数在多个子程序间传递，这样可能表面应该把这些子程序重构到一个类里，把这个参数当成对象数据共享。实质上让参数传递得更顺畅不是目标，但把大量数据到处传递是在暗示换一种类的组织方式更好。
    - 创建中心控制点
    - 让代码更易于重用
    - 为程序族做计划
    - 把相关操作放到一起
    - 实现特定的重构

  - 应该避免的类
    - 避免创建万能类
    - 消除无关紧要的类
    - 避免用动词命名类

## 第七章 高质量的子程序
### 7.1 创建子程序的正当理由
创建子程序的正当理由（可能有重叠）：  
  - 降低复杂度  
    隐藏信息、缩小代码规模改善可维护性，总之就是子程序可以有很好的抽象能力。当条件语句或者循环嵌套过深时就该用子程序提取来降低复杂度了。
  - 引入中间、易懂的抽象  
    把一段代码放入一个命名恰当的子程序内是说明这段代码最好的方式之一。给这个方法起个好名字，然后这个方法名提供了更高层次的抽象，提高易读性的同时降低了复杂度。
    ```个人理解
    一个简单的功能单一的方法，容易抽象出一个能够自解释的方法名，自解释意味着抽象程度高，自解释的方法名比注释、文档这些东西好多了。
    ```
  - 避免代码重复
    ```个人理解
    把散落在类里的起相同作用的代码收集起来放入一个子程序，一是提高了内聚性增加了复用，二是只用修改一处代码方便维护修改。
    ```
  - 支持子类化  
    如果能让virtual（可覆盖）子程序保持简单，在实现派生类时会减少犯错几率。
  - 隐藏顺序
    ```个人理解
    把处理事件的顺序隐藏起来而不是让它散布在系统中，实际上就是强制保证了这个操作的顺序。这种前后有强关联的调用，通过一层子程序的封装就能够使得其他地方不需要担心调用时出现差错，不必做额外的处理保证顺序。
    ```
  - 隐藏指针操作  
    指针可读性差且容易出错。隔离在程序内部同样能够提高正确性，且将修改时的影响限制在子程序内。
  - 提高可移植性  
    用于隔离程序中不可移植部分，从而明确识别和隔离未来移植工作。不可移植的部分包括编程语言提供的非标准功能、对硬件的依赖、对操作系统的依赖。
  - 简化复杂的bool判断  
    理解程序的流程时通常没有必要了解判断的实现细节，且判断的目的容易用一个方法名概括。这也是为了提升可读性，能够更容易理清代码流程和降低程序复杂度。
  - 改善性能  
    方便查出运行效率低下的代码（提高可读性，降低复杂度），在一处进行优化时全部调用的地方都会收益（内聚性的体现）。
  - 确保所有子程序都很小  
    这一点是没有必要的，事实上，有些事情写个大的子程序来完成会更好。
    ```个人想法
    在看到书上否认这一点前，实际上我常常因此痛苦，因为我此前一直认为任何情况下子程序精简到单一功能自解释都是好的应尽量这么做，但是实际开发中真的这么做了就会发现很难把握这个度，绝大多数情况下都是过度设计，现在感觉可以不必纠结了，本书接下来的内容里有对“有些事情写个大的子程序完成会更好”的阐述。
    ```
  - 创建子程序的理由很多地方和创建类理由不谋而合
    隔离复杂度、隐藏实现细节、限制变化带来的影响、隐藏全局数据、形成中央控制点、促成可重用代码、达成特定重构目的。

似乎过于简单而没必要写成子程序的操作  
不要避讳去为一个简单目的写一个简单的子程序。一个甚至只有的很好又小巧的两三行代码的子程序实际上很有用。
- 提高可读性  
  例如把某个简短的公式封装起来，起个恰当的名字，相比于把这个公式到处列出来每次都要看懂这个公式，一个自解释名字的方法的可读性明显更高。
- 提高可扩展性  
  把简单的操作封装起来，因为往往简单的操作会因为各种方面的变更和累积变成复杂操作，这样只用增加一点代码就够了。
  ```个人感受
  提前意识到这件事会舒服很多，真的要改变时再提取出来写好，再每处逐一变更真的很痛苦，而且总是要担心这次修改会不会导致一些错误，我在这个地方实际上已经吃过很多次的亏了。
  ```

### 7.2 在子程序层上设计
内聚性的概念通常在类这一层次的设计中更为常用。对子程序而言，内聚性是指子程序中各种操作之间联系的紧密程度。  
我们的目标是让每一个子程序只把一件事做好，不再做其他事。这种做法的好处是子程序有更高的可靠性。  

内聚性的几个层次的相关概念  
这些是可取的内聚性，是可以使用的：
- 功能的内聚性
  最强也最好的一种内聚性，让一个子程序仅执行一项操作（前提是所执行操作与方法名相符）。
- 顺序上的内聚性  
  子程序内包含有需要按特定顺序执行的操作，且这些步骤需要共享数据，而且只有全部步骤执行完毕后才完成一项完整的功能（不能拆开用其中一个步骤单独执行完成功能）。
  ```个人理解
  相比于功能的内聚性，顺序上的内聚性是完成一个功能，并且包含该功能的所有操作。
  ```
- 通信上的内聚性  
  子程序内的不同操作使用同样的数据，但操作间不存在任何联系。
  ```个人理解
  相比于顺序上的内聚性，这次更松散了，不同的操作不必一同完成同一个功能。
  ```
- 临时的内聚性
  需要同时执行才放到一起操作的子程序。把临时性的子程序看作某一系列事件的组织者。

而下面这些是不良的内聚性（不应该使用，会导致代码混乱、难于调试不便修改。如果一个程序有不良的内聚性，应该重写使其具有更好的内聚性）：  
- 过程上的内聚性  
  仅仅是把一组操作赋以特定的顺序，除此之外这些操作不需要为了的任何原因而彼此关联。
- 逻辑上的内聚性  
  若干操作被放入同一个子程序中，通过传入的控制标志选择执行其中一项操作。控制流是将这些操作放到一起的唯一原因（通常被包在一个很大的if...else语句或switch...case语句中）。  
  PS:标志是各项操作间没有任何关联。若仅由一系列if-else和switch-case语句组成，此时如果唯一作用是发布各种命令而不处理任何命令，这样一个逻辑上内聚的子程序也有可取之处，即充当事件管理器。
- 巧合的内聚性  
  指子程序的各个操作间没有任何可以看到的关联，即“无内聚性”或“混乱的内聚性”。

### 7.3 好的子程序的名字
- 描述子程序所做的所有事情  
  子程序的名字应当描述其所有的输出结果以及副作用（如果有）。
  但是描述太长很繁冗会显得很蠢时，解决方法并不是使用描述性较弱的程序名，而是换一种方式编写程序，直到能够在较短长度是准确解释。
- 避免使用无意义的、模糊或表述不清的动词
